# AGENTS.md

Guidance for future coding agents working on Aggasys Gray Bot.

## Project Shape

- Gray is an Aggasys Telegram operations bot with a Hermes-style control harness.
- The bot is intended to behave like an internal AI teammate for SME operations, but with stronger guardrails than a free-form chat agent.
- Hermes is the safety/control layer: policy decisions, audit logging, approvals, role checks, scheduler jobs, standups, and operational safeguards.
- Supply ordering is intentionally not implemented yet. Do not add spend/order workflows unless explicitly requested and guarded behind approval.

## Current Runtime Assumptions

- Production deploy target is the self-hosted VM runner for GitHub repo `Slyriss/Aggasys-Gray-Bot`.
- Production model mode is DeepSeek API mode:
  - `MODEL_PROVIDER=deepseek`
  - `DEEPSEEK_API_KEY` comes from GitHub Actions secrets or the VM `.env`.
  - `DEEPSEEK_BASE_URL=https://api.deepseek.com`
  - `DEEPSEEK_MODEL=deepseek-v4-flash`
  - `EMBEDDING_PROVIDER=disabled`
- Ollama is not required when DeepSeek mode is active. Do not assume local model hosting is needed.
- `DEPLOY_STATUS.md` is generated by the VM deployment workflow and committed back with `[skip ci]`.

## Deployment Flow

- Normal deployment happens by pushing to `main`; `.github/workflows/deploy.yml` runs on the self-hosted runner.
- The workflow syncs GitHub secrets into the VM `.env`, runs verification gates, migrates the database, rebuilds/restarts the bot, performs health checks, and publishes `DEPLOY_STATUS.md`.
- If a GitHub secret such as `DEEPSEEK_API_KEY` is updated without a code change, rerun the latest deploy workflow to sync it into the VM.
- After any deployment push or rerun, verify with:
  - `gh run list --repo Slyriss/Aggasys-Gray-Bot --workflow deploy.yml --limit 5`
  - `gh run watch <run_id> --repo Slyriss/Aggasys-Gray-Bot --exit-status`
  - `git pull --ff-only origin main`
  - `python scripts\verify_deploy_status.py DEPLOY_STATUS.md`
- A healthy deploy status should show the target commit, `Application started`, `MODEL_PROVIDER=deepseek`, and `Post-deploy health check OK`.

## Required Local Checks

Before committing or pushing runtime changes, run:

```powershell
python scripts\run_checks.py
```

This covers unit tests, compile checks, secret hygiene, runtime import smoke, deployment asset verification, runtime/schema/policy verification, command surface verification, and deploy status verification.

Focused checks used often:

```powershell
python -m unittest discover -s tests -p "test_bot_commands.py"
python scripts\verify_deployment_assets.py
python scripts\verify_deploy_status.py DEPLOY_STATUS.md
```

## Security And Privacy Lessons

- Never paste, commit, echo, or log real secrets. DeepSeek keys belong in GitHub Secrets or VM `.env`, not in chat or source.
- Startup logs must not print raw Telegram user IDs for `ALLOWED_USERS`, `ADMIN_USERS`, or `OPERATOR_USERS`; log counts only.
- Agent operational logs must avoid raw user content, raw tool output, full URLs, query strings, credentials, and exception messages.
- Use `bot/log_safety.py` helpers for logging metadata only:
  - exception type instead of exception message
  - URL host instead of full URL
  - text/result length instead of content snippets
- `scripts/scan_secret_hygiene.py` is part of the safety gate and should stay strict.
- `DEPLOY_STATUS.md` must remain redacted and must pass `scripts/verify_deploy_status.py`.

## Telegram Command Safety

- Personal-data commands only run in private chat:
  - `/recall`
  - `/memory`
  - `/tasks`
  - `/brief`
  - `/forget_me`
- If a user tries those in a group chat, Gray should refuse with a private-chat instruction and audit the refusal as `blocked_private_chat`.
- Operator/admin workflow commands can remain group-friendly but must respect role checks.
- `/ops_status` is admin-only and should show redacted runtime state, never secret values.

## Role Model

- `ALLOWED_USERS`: users who may chat with Gray and use personal/read commands.
- `OPERATOR_USERS`: users who may run team workflow commands such as standups and schedules.
- `ADMIN_USERS`: users who may approve/deny Hermes requests, schedule web monitors, ingest/lint company knowledge, and run ops status.
- Admins are operators automatically.
- Preflight requires `ADMIN_USERS` and `OPERATOR_USERS` to be subsets of `ALLOWED_USERS`.

## Standup And Scheduler Lessons

- Standups should be scheduled once and then auto-triggered; users should not need to prompt daily standups manually.
- Duplicate recurring schedules should be prevented.
- Daily times should be normalized, including compact `HHMM` input.
- Standup summary recipients are tiered as `chat`, `admins`, or `both`.
- Admin DM failures should not make the entire standup summary job fail; record failed admin deliveries in audit details.
- Scheduler failures should auto-pause after the configured failure threshold and audit `scheduled_job_auto_paused`.

## Data Retention And Operations

- Hermes audit and operational retention are controlled by:
  - `HERMES_AUDIT_RETENTION_DAYS`
  - `HERMES_OPERATION_RETENTION_DAYS`
- Keep backup/restore scripts and retention pruning covered by verifiers.
- Use strict migrations with `psql -v ON_ERROR_STOP=1`.
- Database pool and model/embedding clients must close during shutdown.

## Testing Lessons

- Tests may stub modules like `telegram`, `tools`, `url_ingester`, and `model_client`; restore `sys.modules` after isolated tests to avoid poisoning later tests.
- Prefer focused regression tests for each safety rule and add asset-verifier markers when a guardrail must not disappear.
- If a test command fails because the `tests` directory is not a Python package, use unittest discovery:

```powershell
python -m unittest discover -s tests -p "test_name.py"
```

## Git And Worktree Habits

- The worktree may be dirty from user or generated changes; inspect before editing and never revert unrelated work.
- Use `apply_patch` for manual file edits.
- Commit small hardening increments with clear messages.
- After successful stage/commit/push actions, final responses should include the Codex git directives required by the desktop app.

## Current Enterprise-Readiness Direction

Continue improving Gray toward company use by prioritizing:

- secret and log hygiene
- RBAC and private-data boundaries
- audited refusals and failures
- deploy health and rollback confidence
- scheduler correctness
- clear operator/admin documentation
- tests and verifiers that prevent safety regressions
